<div className="challenges">

**모노레포 단방향 의존 아키텍처 + 공통 인프라 패키지화**

- 문제: 공통 인프라(인증, API, 상태관리, i18n, UI)가 앱별로 중복 구현되고 컨벤션이 분산되어 신규 앱 추가 시 기본 설정에만 2~3일이 소요되는 비효율이 있었고, 패키지 간 의존 관계 통제가 없어 apps → packages 구조가 깨지거나 잘못된 소속·역참조가 발생할 수 있는 구조적 위험이 있었습니다.
- 해결:
  - 공통 인프라를 패키지 단위로 추상화해 앱과 분리하고, 모든 앱이 동일한 기반 위에서 동작하도록 구조를 재정의했습니다. 앱 템플릿과 생성 스크립트를 도입해 공통 인프라를 자동화했습니다.
  - 아키텍처 및 개발 규칙을 문서화하고, AI 에이전트가 코드 작성·리뷰 단계에서 해당 규칙을 일관되게 참조·적용하도록 가이딩 체계를 구축했습니다.
- 성과:
  - **신규 앱 셋업 시간을 2~3일에서 30분 이내로 단축**하여 초기 개발 비용을 절감했습니다.
  - 공통 인프라 재사용으로 **앱 간 구현 편차를 제거**하고, **개발 경험의 일관성을 확보**했습니다.
  - 의존 규칙 문서화 + AI 에이전트 가이딩으로, **의존성 위반에 대한 사람 리뷰 의존도를 낮추고 구조 일관성을 유지**했습니다.

**SPA 공통 인증·인가 인프라 재설계**

- 문제: 모노레포 환경에서 여러 SPA가 안전하고 일관되게 사용할 수 있는 인증·인가 시스템이 필요했습니다. 특히 XSS 공격을 원천 차단해야 했고, 토큰 만료 시 발생하는 인터셉터 내 무한 루프나 API 동시 요청에 따른 레이스 컨디션 등의 병목 문제를 방지할 수 있는 아키텍처 설계가 필요했습니다.
- 해결:
  - HttpOnly 쿠키 인증 체계를 도입하여 시스템을 일원화하고, 서버 데이터를 기반으로 프론트엔드 화면 및 기능 접근을 안전하게 제어하는 RBAC 시스템을 구축했습니다.
  - 재발급 로직을 Axios 인터셉터 체인에서 분리하여 순환 호출을 차단하고, Spring Cloud Gateway 에러 코드를 기반으로 꼭 필요한 상황에서만 조건부로 재발급을 수행하도록 구조를 개편했습니다.
  - @repo/api-client(공통 API)와 @repo/auth(인증/인가)를 독립된 패키지로 분리하여 코드 재사용성 극대화.
- 성과:
  - 모노레포 내 여러 SPA의 인증 및 RBAC 권한 제어 로직을 공통 패키지로 일원화하여 **중복 코드를 제거하고 공통 인증 규약을 수립**했습니다.
  - 토큰 재발급 시의 **무한 루프와 API 중복 호출 문제를 해결**하고, XSS 공격을 방어하는 HttpOnly 쿠키 인증 시스템을 통해 안정성을 확보했습니다.

<div className="challenges">

**주문 라이프사이클 디지털 전환**

- 문제: 주문 등록부터 문서 발행, 출고 송장 작성까지 수기 작성에 의존하는 구조였습니다. 이로 인해 신규 주문 생성을 별도 처리해야 했으며, 시스템과 문서 간 데이터가 어긋나고 오발송·누락·발행 이력 추적 불가의 구조적 한계가 발생했습니다.
- 해결:
  - **엑셀 업로드 기반 다건 품목 등록**과 주문 분할 시 **원본·신규 패널 실시간 업데이트**로 클라이언트 측 입력 흐름을 **일괄 처리·자동 정합화 구조**로 전환했습니다.
  - 송장·견적서 생성·수정·발행을 시스템 내에서 처리하는 파이프라인을 구축해 외부 의존을 제거하고, 발행 이력을 DB 기준으로 관리해 추적·재발행이 가능하도록 구조화했습니다.
  - 영업 담당자가 별도 문의 없이 송장번호·배송현황을 직접 조회할 수 있도록 인터페이스를 구현했습니다.
- 성과:
  - 주문 분할 시 **수량 불일치 이슈 0건**, **문서 발행 이력 추적·재발행 지원**
  - 송장 발행: **연 약 41시간 절감** (연 500건 × 건당 5분 기준 산출)
  - 택배 송장 관리: **연 약 106시간 절감** (월 755.5분 → 226.6분, **약 70% 감소** 실측)

**성과/보상 연결 평가 시스템 구축 — 전략 성과 정량화 및 의사결정 구조 전환**

- 문제: OKR 성과 데이터가 존재했으나, 기여도를 정량화할 기준이 없어서 성과 → 평가 → 보상 간 연결이 단절되어 있었고, 개인을 평가하는데 올바른 근거를 찾기가 어려웠습니다.
- 해결:
  - 데이터 기반 점수 산출 모델을 설계하고, 점수 → 비중 → 보상으로 이어지는 계산 구조를 구현했습니다.
    OKR 성과 피드백 → CEO Review → 보상 확정까지 이어지는 평가 워크플로우를 구축하고, 정밀 연산 기반 보너스 계산 및 합계 검증 로직을 적용했습니다.
- 성과:
  - 1년간 축적된 OKR 전략 성과 데이터를 **Objective 단위·개인 단위로 통합 리뷰 가능한 메뉴 신규 개발**
  - 정밀 연산 유틸리티 + 예산 합계 실시간 검증으로 성과 산정 시 **합계 불일치·소수점 정밀도 오류 구조적 차단**
