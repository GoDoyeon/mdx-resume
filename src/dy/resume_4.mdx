**[Neuro] 모노레포 공통 패키지 구축 및 의존성 아키텍처 표준화**

- 문제: 공통 인프라(인증, API, 상태관리, i18n, UI)가 앱별로 중복 구현되고 컨벤션이 분산되어 신규 앱 추가 시 기본 설정에만 **1~2일이 소요되는 비효율**이 있었고, 패키지 간 의존 관계 통제가 없어 apps → packages 구조가 깨지거나 잘못된 소속·역참조가 발생할 수 있는 구조적 위험이 있었습니다.
- 해결:
  - 공통 인프라를 **패키지 단위로 추상화해 앱과 분리**하고, 모든 앱이 동일한 기반 위에서 동작하도록 구조를 재정의했습니다. **앱 템플릿과 생성 스크립트**를 도입해 공통 인프라를 자동화했습니다.
  - 아키텍처 및 개발 규칙을 문서화하고, **AI 에이전트가 코드 작성·리뷰 단계에서 해당 규칙을 일관되게 참조·적용**하도록 가이딩 체계를 구축했습니다.
- 성과:
  - **신규 앱 셋업 시간을 1~2일에서 30분 이내로 단축**하여 초기 개발 비용을 절감했습니다.
  - 공통 인프라 재사용으로 **앱 간 구현 편차를 제거**하고, **개발 경험의 일관성을 확보**했습니다.
  - 의존 규칙 문서화 + AI 에이전트 가이딩으로, **의존성 위반에 대한 사람 리뷰 의존도를 낮추고 구조 일관성을 유지**했습니다.

**[Neuro] 보안 및 병목 해결을 위한 공통 인증·인가 모듈 설계**

- 문제: 모노레포 환경에서 여러 SPA가 안전하고 일관되게 사용할 수 있는 인증·인가 시스템이 필요했습니다. 특히 **XSS 공격을 원천 차단함**과 동시에 HttpOnly 쿠키 도입으로 인한 **CSRF 취약점까지 방어**해야 했고, 토큰 만료 시 발생하는 **인터셉터 내 무한 루프**나 API 동시 요청에 따른 **레이스 컨디션 등의 병목 문제를 방지**할 수 있는 아키텍처 설계가 필요했습니다.
- 해결:
  - **HttpOnly 쿠키 인증 체계**를 도입하여 시스템을 일원화하고, 서버 데이터를 기반으로 프론트엔드 화면 및 기능 접근을 안전하게 제어하는 **RBAC 시스템**을 구축했습니다.
  - 재발급 로직을 **Axios 인터셉터 체인에서 분리하여 순환 호출을 차단**하고, Spring Cloud Gateway 에러 코드를 기반으로 꼭 필요한 상황에서만 **조건부로 재발급을 수행**하도록 구조를 개편했습니다.
  - 게이트웨이의 Double Submit Cookie 검증 방식에 대응하기 위해 공통 Axios 인스턴스에 XSRF-TOKEN 쿠키를 **X-CSRF-Token 헤더로 자동 echo** 하도록 설정했습니다.
  - @repo/api-client(공통 API)와 @repo/auth(인증/인가)를 독립된 패키지로 분리하여 코드 재사용성 극대화.
- 성과:
  - 모노레포 내 여러 SPA의 인증 및 RBAC 권한 제어 로직을 공통 패키지로 일원화하여 **중복 코드를 제거하고 공통 인증 규약을 수립**했습니다.
  - 토큰 재발급 시의 **무한 루프와 API 중복 호출 문제를 해결**하고, XSS 공격을 방어하는 HttpOnly 쿠키 인증 시스템을 통해 안정성을 확보했습니다.

  - **차트별 최대 수치 스케일을 통일**하고 유저별 데이터를 한 차트에 겹쳐 볼 수 있도록 설계하여, 왜곡 없이 동료 간 역량을 객관적으로 비교할 수 있도록 구현
- 성과:
